Kafka + Flink score every transaction live. Spark independently retrains on the full history. A reconciliation layer measures exactly where the two disagree — and why.

In plain terms: most systems either respond instantly with incomplete information, or wait for complete information and respond slowly. This project builds both paths side by side, on the same transactions, and measures precisely how much accuracy the fast path gives up for speed — 1.57% disagreement, in this case, and it traces back to one specific, explainable cause below.

Process

How It Was Built — 7 Days

Day 1
EDA + leakage investigation. Found and removed a feature that scored 0.947 AUC alone — traced to a PaySim simulator artifact, not real fraud signal. Locked in a 4-feature model via walk-forward validation.
Day 2
Kafka producer. Time-ordered replay of PaySim, keyed by destination account. Measured 13,316 msg/s sustained throughput; verified per-key ordering held across partitions.
Days 3–4
Flink speed layer. Stateful per-account counter in Flink's ValueState, live XGBoost scoring, checkpointed writes to Delta Lake. The hardest infrastructure stretch of the week — PyFlink's Python↔JVM bridge required resolving a long, specific dependency chain before a single record would flow.
Day 5
Spark batch layer + Hive. Retrained on the full 2,770,409-row historical dataset via Spark window functions, wrote curated predictions to a real Hive Metastore table.
Day 6
Reconciliation. Joined speed vs. batch predictions on a composite key across 24,327 transactions. Found 98.43% agreement and diagnosed the exact mechanism behind the remaining disagreements.
Day 7
Dashboard + this page. Streamlit dashboard for the reconciliation results, architecture diagram, and final documentation.

Analysis

Key Findings

01 / Data Investigation

Leakage Discovered and Eliminated

An early feature (orig_balance_error) achieved 0.947 AUC alone — a red flag. Root cause: 32.9% of legit transactions have zero-value balances in PaySim, while 99.5% of fraud shows perfect balance reconciliation. A simulator artifact, not a real signal. Dropped.

02 / Distribution Shift

Non-Obvious Temporal Drift

PaySim's legit volume collapses after step ~400 while absolute fraud count stays flat — fraud rate varies 20×+ across the timeline. Addressed with walk-forward validation across 5 temporal folds rather than a single random split.

03 / Model Performance

Consistent Across Both Engines

XGBoost, 4 features. Walk-forward: AUC 0.9995 ± 0.0003, PR-AUC 0.940 ± 0.049. Batch-retrained (Day 5): AUC 0.9996, PR-AUC 0.957 — confirming reproducibility across pandas/Flink vs Spark execution paths.

04 / Reconciliation

Cold-Start Explains the Gap

Speed layer: PR-AUC 0.8475, 100% recall. Batch layer: PR-AUC 0.8187, missed 1 fraud case the speed layer caught. The 1.57% disagreement traces almost entirely to dest_txn_count_so_far — Flink's state starts at 0 on startup, Spark has complete history. An architectural tradeoff, not a bug.

Architecture

Design Decisions & Tradeoffs

DecisionReasoning
Filter to TRANSFER / CASH_OUT 100% of fraud lives in these two types. Drops stream volume ~56% with zero recall loss — reduces state footprint and compute load in Flink before any scoring happens.
Processing-time, no watermarks Kafka's per-key ordering guarantee (keyed by nameDest) was sufficient for this project's correctness needs. Event-time watermarking is a documented production gap, not an oversight.
Flink local execution mode The official Flink Docker image lacks Python/JDK-dev headers for PyFlink's native dependencies (pemja). Local mode is a standard, documented PyFlink execution mode — not a workaround.
Flink → staged files → Spark → Delta Delta's write path is Spark-native. Staging decouples the two runtimes cleanly rather than loading a full Spark session inside Flink's per-record hot path.
Composite txn_key Flink's live stream and the batch CSV share no natural identifier. nameOrig|nameDest|step|amount gives both sides an independently derivable join key from data each already has.
Delta MERGE over blind append Makes the staging-to-Delta commit idempotent — safe to re-run over the same staged files without duplicating records. ACID semantics as a side-effect.

Honesty

Known Limitations

  • txn_key uniqueness is not formally guaranteed — collisions possible with identical nameOrig/nameDest/step/amount. Acceptable for PaySim's characteristics, not production-grade.
  • Float formatting between Python (Flink) and Spark for the amount component of txn_key was spot-checked, not exhaustively verified across edge cases.
  • Hive Metastore path is bind-mounted to an absolute local path — would need S3-backed storage for portability.
  • Sample size (24,327 joined transactions, 28 fraud cases) is adequate for directional conclusions but small by rigorous statistical standards.
  • No event-time watermarking — acceptable given Kafka's ordering guarantees here, but a real production requirement for out-of-order tolerance.
